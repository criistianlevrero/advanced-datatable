## UC-01 — Edit Single Cell

**Actor:** User  
**Trigger:** User changes the value of a cell in the grid  
**Preconditions:**
- Table is loaded
- Target cell exists (row and column present in schema)

**Input:**
```
rowId: string
colId: string
value: any
```

**Main Flow:**
1. UI captures the new value from the cell editor input
2. UI creates a `SetCellOperation` with a client-generated `id`, `type: "set_cell"`, `rowId`, `colId`, `value`, `source: "client"`, `status: "pending"`, and `target: { type: "cell", rowId, colId }`
3. UI calls `store.dispatch(op)` → `manager.apply(op)`
4. OperationManager registers op as `pending`, indexes target
5. Engine applies op optimistically: upserts cell in `row.cells[colId]`
6. Store notifies Zustand subscribers → UI re-renders with new value + loading indicator
7. OperationManager batches op and sends via `transport.send([op])` after debounce window
8. Backend validates op, persists, returns `{ ack: ["op1"] }`
9. OperationManager calls `confirm("op1")` → removes from pending index
10. UI re-renders: loading indicator disappears

**Output:**
- Cell displays new value
- No loading state on cell
- Operation no longer in pending set

**Edge Cases:**
- EC-01a: Backend rejects → see UC-09
- EC-01b: Network error → see UC-14
- EC-01c: Column does not exist → engine buffers or ignores per policy (see REQ-023)
- EC-01d: Row does not exist → engine upserts row (create-on-write policy)

## UC-02 — Paste Bulk Data (Range)

**Actor:** User  
**Trigger:** User pastes clipboard data covering multiple cells (e.g., 3×3 grid)  
**Preconditions:**
- Table is loaded
- All target rows and columns exist in schema

**Input:**
```
updates: Array<{ rowId: string; colId: string; value: any }>
targetRange: { rowIds: string[]; colIds: string[] }
```

**Main Flow:**
1. UI intercepts clipboard paste event
2. UI parses clipboard content into a 2D array and maps to row/column IDs
3. UI creates a `BulkUpdateOperation` with all updates and `target: { type: "range", rowIds, colIds }`
4. Store dispatches → OperationManager registers and applies optimistically
5. Engine iterates updates; calls internal `applySetCell` for each
6. Store triggers re-render: entire range shows new values + loading indicators
7. OperationManager batches and sends op
8. Backend confirms → loading removed from range

**Output:**
- All pasted cells display new values
- Entire range loading indicator cleared

**Edge Cases:**
- EC-02a: Some cells in range don't exist → engine creates rows on write
- EC-02b: Backend partially rejects (some ops fail, some ack) → OperationManager handles per-op status
- EC-02c: Clipboard data has more columns than schema → those cells are ignored by UI before creating op

## UC-03 — Receive Server Column Update

**Actor:** Server  
**Trigger:** Backend sends a derived column recalculation (e.g., "total" column)  
**Preconditions:**
- Column `colId` exists in schema
- Transport layer has received the server push (WebSocket or polling response)

**Input:**
```ts
serverOp: {
  id: "op3",
  type: "update_column",
  colId: "total",
  values: { r1: 100, r2: 200 },
  source: "server"
}
```

**Main Flow:**
1. Transport layer receives server operation
2. OperationManager calls `engine.apply(serverOp)`
3. Engine iterates `values`; for each `rowId`, calls internal `applySetCell`
4. Store Zustand subscription fires → UI re-renders affected cells
5. No loading state is applied (server op is already authoritative)

**Output:**
- Affected cells display updated values
- No loading state shown (source is server)

**Edge Cases:**
- EC-03a: `colId` does not exist in schema → operation is buffered until `add_column` op arrives
- EC-03b: Some `rowId` values reference non-existent rows → engine upserts rows

## UC-04 — Add Column Dynamically

**Actor:** Server / Admin  
**Trigger:** Backend emits an `add_column` schema operation  
**Preconditions:**
- Column ID does not currently exist in schema

**Input:**
```ts
schemaOp: {
  id: "op4",
  type: "add_column",
  column: { id: "discount", type: "number" },
  index: 2,
  source: "server"
}
```

**Main Flow:**
1. Server sends `add_column` operation via transport
2. OperationManager applies via `engine.apply(schemaOp)`
3. Engine adds column to `schema.columns`, inserts into `columnOrder` at `index`, increments `schema.version`
4. Store notifies → UI renders new empty column
5. Subsequent `update_column` or `set_cell` ops populate cells progressively

**Output:**
- New column appears in grid at specified position
- All cells in new column display default value (null or column default)
- `schema.version` incremented

**Edge Cases:**
- EC-04a: `column.id` already exists → operation is no-op (idempotence)
- EC-04b: `index` is out of range → column is appended
- EC-04c: Client-side user also tries to add column with same ID simultaneously → conflict deferred to reconciliation

## UC-05 — Remove Column

**Actor:** User / Server  
**Trigger:** User or server removes a column from the table  
**Preconditions:**
- Column `columnId` exists in schema

**Input:**
```
columnId: string
```

**Main Flow:**
1. Create `RemoveColumnOperation` for `columnId`
2. Apply optimistically: remove from `schema.columns`, remove from `columnOrder`, increment `schema.version`
3. UI removes column from grid
4. Existing cell data for that column is NOT deleted from rows (lazy cleanup)
5. Backend confirms

**Output:**
- Column disappears from grid
- `schema.version` incremented

**Edge Cases:**
- EC-05a: Column does not exist → operation is no-op

## UC-06 — Add Row

**Actor:** User / Server  
**Trigger:** User clicks "Add row" or server pushes a new row  
**Preconditions:**
- Row ID does not already exist in rows

**Input:**
```
row: Row   // { id: string; cells: {} }
index?: number
```

**Main Flow:**
1. Create `AddRowOperation` with new `Row` object
2. Apply optimistically: `rows.set(row.id, row)`, `insertIntoOrder(rowOrder, row.id, index)`
3. Store triggers re-render: new empty row appears at correct position
4. Backend confirms

**Output:**
- New row appears in grid at specified position
- Row is empty (no cells populated)

**Edge Cases:**
- EC-06a: Row ID already exists → operation is no-op (idempotence)
- EC-06b: Index out of range → row is appended

---

## UC-07 — Remove Row

**Actor:** User  
**Trigger:** User deletes a selected row  
**Preconditions:**
- Row `rowId` exists in rows

**Input:**
```
rowId: string
```

**Main Flow:**
1. Create `RemoveRowOperation`
2. Apply optimistically: `rows.delete(rowId)`, `removeFromOrder(rowOrder, rowId)`
3. UI removes row from grid
4. Backend confirms

**Output:**
- Row no longer appears in grid

**Edge Cases:**
- EC-07a: Row does not exist → no-op

## UC-08 — Reorder Columns

**Actor:** User  
**Trigger:** User drags a column header to a new position  
**Preconditions:**
- All column IDs in new order exist in schema

**Input:**
```
columnOrder: string[]   // Complete new order (permutation of existing IDs)
```

**Main Flow:**
1. UI captures drag event; computes new `columnOrder`
2. Creates `ReorderColumnsOperation`
3. Apply: replace `schema.columnOrder` with new order, increment `schema.version`
4. UI re-renders columns in new order

**Output:**
- Columns displayed in new order

**Edge Cases:**
- EC-08a: New order is missing a column ID → operation rejected (validation)
- EC-08b: New order contains unknown IDs → operation rejected

## UC-09 — Operation Rejected by Backend

**Actor:** System  
**Trigger:** Backend returns an error for one or more operations  
**Preconditions:**
- One or more operations are in `applying` state

**Input:**
```
errors: [{ opId: string; reason: string; code?: string }]
```

**Main Flow:**
1. Transport receives `{ errors: [{ opId: "op1", reason: "Invalid value" }] }`
2. OperationManager calls `fail("op1", error)`
3. Operation transitions to `error` state
4. Engine optionally reverts the optimistic change (revert to previous value)
5. TargetIndex updates: pending set for target is cleared
6. UI re-renders: error indicator shown on target cell/range

**Output:**
- Cell/range shows error state
- Value may be reverted to pre-operation state

**Edge Cases:**
- EC-09a: Partial rejection (some ops ack, some error) → handle per-op
- EC-09b: Engine revert is not implemented → cell keeps optimistic value with error indicator
- EC-09c: User corrects and resubmits → new op replaces error state

## UC-10 — Initial Table Load

**Actor:** System  
**Trigger:** Table component mounts; state must be initialized from backend  
**Preconditions:**
- Network is available
- Backend endpoint `GET /table` is available

**Input:** (none)

**Main Flow:**
1. Store initializes with empty `TableState`
2. Transport sends `GET /table`
3. Backend returns `{ schema, rows, rowOrder, version }`
4. Store hydrates: constructs `TableState` from response (wire format → engine format)
5. Engine loads schema and rows into memory
6. UI renders initial grid (with virtualization active)

**Output:**
- Table displays initial data
- Schema version matches backend

**Edge Cases:**
- EC-10a: Network error → UI shows loading error state; retry can be triggered
- EC-10b: Response is empty (no data) → engine initialized with empty state

## UC-11 — Insert Rows While Scrolling (Scroll Stability)

**Actor:** User / Server  
**Trigger:** Row(s) are inserted while the user has the table scrolled to a non-zero position  
**Preconditions:**
- User is scrolled past row 0
- An `add_row` operation is applied (client or server)

**Main Flow:**
1. UI captures current scroll anchor (first fully visible row ID)
2. `add_row` operation is applied → `rowOrder` changes
3. Virtualizer recomputes layout
4. UI restores scroll position to the anchored row

**Output:**
- Table content shifts correctly
- Viewport does not jump; user sees same rows as before insertion

**Edge Cases:**
- EC-11a: Anchor row was deleted → scroll to nearest neighbor
- EC-11b: Multiple rows inserted simultaneously → single recompute with single restore

## UC-12 — Receive Server Correction (Reconciliation)

**Actor:** Server  
**Trigger:** Backend sends `serverOps` in response to a client operation  
**Preconditions:**
- Client-sent operation was in `applying` state
- Backend computed a correction or transformation

**Input:**
```ts
response: {
  ack: ["op1"],
  serverOps: [{
    id: "op_server_1",
    type: "set_cell",
    rowId: "r1",
    colId: "A",
    value: 99,   // Server corrected value
    source: "server"
  }]
}
```

**Main Flow:**
1. OperationManager confirms `op1` (removes from pending)
2. OperationManager calls `engine.apply(serverOp)` for each server op
3. Engine overwrites the optimistic value with server's authoritative value
4. UI re-renders with corrected value; no loading state

**Output:**
- Cell shows server's corrected value
- No loading indicator

## UC-13 — Server-Initiated Operations via Streaming (Post-MVP)

**Actor:** Server  
**Trigger:** Backend pushes operations via WebSocket (e.g., another user made a change)  
**Preconditions:**
- WebSocket connection is active
- Transport uses `WebSocketTransport` adapter

**Main Flow:**
1. Transport `subscribe` callback receives `{ type: "ops", ops: Operation[] }`
2. OperationManager calls `engine.applyBatch(ops)` for all received ops
3. Store triggers re-render → UI reflects remote changes

**Output:**
- Real-time updates from other users or server reflected in UI

---

## UC-14 — Network Error on Operation Submit

**Actor:** System  
**Trigger:** Transport layer fails to deliver operations to backend  
**Preconditions:**
- One or more operations are in `applying` state

**Main Flow:**
1. `transport.send()` rejects (network timeout / HTTP 5xx)
2. OperationManager catches the error
3. Operations remain in `pending` / `applying` state
4. OperationManager queues for retry (configurable retry policy)
5. After max retries exhausted → `fail(opId)` called for all affected ops

**Output:**
- UI shows error/retry state for affected cells
- User can manually retry or discard

**Edge Cases:**
- EC-14a: Backend received ops but response was lost → backend idempotence prevents duplicates on retry (same `op.id`)
