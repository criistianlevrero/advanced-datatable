The implementation is organized into **stages** (high-level milestones) and **units** (individual files to implement). Each unit is the smallest independently implementable artifact. Units are ordered by dependency: no unit depends on a unit listed after it in the same stage.

**Convention for unit IDs:** `STAGE-PACKAGE-FILE`  
e.g., `S1-CORE-01` = Stage 1, package `core`, first file.

## Stage 4 ‚Äî TableEngine Implementation

| Unit | File | Depends On | Priority |
|------|------|-----------|----------|
| S4-CORE-01 | `packages/core/src/engine/ITableEngine.ts` | S1-CORE-04, S1-CORE-05 | Critical |

| S4-CORE-03 | `packages/core/src/index.ts` | S4-CORE-01, S4-CORE-02, S1-CORE-06 | Critical |

**`TableEngineImpl` behavior:**
- Holds `TableState` internally (schema + rows Map + rowOrder)
- Maintains `Set<string>` of applied op IDs (idempotence dedup)
- Dispatches `apply(op)` to correct strategy based on `op.type`
- `applyBatch(ops)` = sequential `apply()` calls

**Stage 4 deliverable:** Fully functional, testable `TableEngineImpl` with no external dependencies.

## Stage 5 ‚Äî Operation Layer

| Unit | File | Depends On | Priority |
|------|------|-----------|----------|
| S5-OPS-01 | `packages/operations/src/types.ts` | S1-CORE-06 | Critical |
| S5-OPS-02 | `packages/operations/src/targetKey.ts` | S1-CORE-05 | Critical |
| S5-OPS-03 | `packages/operations/src/ITargetIndex.ts` | S1-CORE-05 | High |
| S5-OPS-04 | `packages/operations/src/TargetIndexImpl.ts` | S5-OPS-02, S5-OPS-03 | High |
| S5-OPS-05 | `packages/operations/src/IOperationBatcher.ts` | S1-CORE-05 | High |
| S5-OPS-06 | `packages/operations/src/OperationBatcherImpl.ts` | S5-OPS-05 | High |
| S5-OPS-07 | `packages/operations/src/IOperationManager.ts` | S1-CORE-05, S5-OPS-01 | Critical |
| S5-OPS-08 | `packages/operations/src/OperationManagerImpl.ts` | S5-OPS-01..07, S4-CORE-01 | Critical |
| S5-OPS-09 | `packages/operations/src/index.ts` | All S5-OPS-* | Critical |

**`OperationManagerImpl` behavior:**
- Accepts `ITableEngine` and `IOperationTransport` via constructor
- `apply(op)`: registers op in `records` map, calls `engine.apply(op)`, enqueues to batcher, indexes target
- `confirm(opId)`: transitions to `confirmed`, removes from target index
- `fail(opId, error)`: transitions to `error`, removes from target index, optionally reverts via engine
- `getPendingByTarget(target)`: delegates to `TargetIndexImpl`

**`OperationBatcherImpl` behavior:**
- Buffers ops in a queue
- Starts debounce timer (50ms default) on first enqueue
- On timer fire: calls `transport.send(queue)`, processes response (calls `confirm`/`fail` per result)
- `flush()`: cancels timer, sends immediately
