## 1. Executive Summary

The goal is to build a **reusable DataTable library** implemented as a monorepo. The library exposes packages that can be consumed independently in any TypeScript/React project.

The core design principle is **operation-driven state**: the table state is never mutated directly; all changes flow through typed operations (deltas). This enables optimistic UI updates, batching, undo/redo, backend synchronization, and future real-time collaboration.

The system is structured in layers:

```
UI Layer (React)     → rendering + user interaction
View Model (hooks)   → derived state, selectors
State Layer (Zustand)→ reactive store
Operation Layer      → batching, pending tracking, reconciliation
Table Engine (core)  → deterministic state mutations
Data Source          → API / WebSocket / mock
```

The library is designed to handle datasets up to **500k rows** and **200 columns** through mandatory virtualization, and to scale toward streaming, formulas, and multi-user collaboration in future phases.

## 2. Glossary

| Term | Definition |
|------|-----------|
| **Operation** | An atomic, typed delta that describes a single state change (e.g., `set_cell`, `add_column`). |
| **TableEngine** | The core module responsible for applying operations and maintaining table state deterministically. |
| **OperationManager** | The module responsible for tracking operation lifecycle, batching, retries, and reconciliation. |
| **Schema** | The structural definition of the table: columns with their types, order, and metadata. |
| **Row** | A data record identified by a stable string ID, containing a map of cells keyed by column ID. |
| **Cell** | The smallest unit of data, containing a `value` and optional `meta` map. |
| **TargetDescriptor** | A descriptor that identifies the scope of an operation (cell, row, column, range) for deriving UI loading state. |
| **Optimistic update** | Applying an operation immediately on the client before backend confirmation. |
| **Reconciliation** | The process of merging backend confirmation/corrections with the client-applied optimistic state. |
| **Virtualization** | Rendering only the visible portion of a large dataset to maintain performance. |
| **Pending operation** | An operation applied client-side but not yet confirmed by the backend. |
| **Idempotence** | Property ensuring that applying the same operation multiple times (same `id`) produces no duplicate effects. |
| **Monorepo** | A repository containing multiple independent but related packages managed together. |
| **OperationTransport** | Abstraction interface for network communication (HTTP or WebSocket). Implemented by adapters. |
| **Playground** | A development app inside the monorepo used for manual testing, showcase, and demos. |
| **Data Version** | Optional monotonic version number tracking data mutations for conflict detection. |
| **Schema Version** | Monotonic integer incremented on every structural (schema) change. |
| **Ghost entity** | A column or row referenced in an operation before it is formally registered in the schema. |
| **Anchor (scroll)** | A reference row/cell used to preserve scroll position when structural changes happen. |

## 3. Normalized Requirements

### REQ-001 — Operation-Driven Architecture
**Source:** RFC §3 Principles, §4.4  
**Description:** All state mutations must be expressed as typed operations. Direct state mutation is prohibited. Every operation must have a stable unique `id`, a `type`, a `source` (client/server), and an optional `target` descriptor.  
**Priority:** Critical

---

### REQ-002 — UI Decoupling
**Source:** RFC §3 Principles  
**Description:** The core `TableEngine` must have zero dependencies on React or any UI framework. It must be testable in isolation in a Node.js environment.  
**Priority:** Critical

---

### REQ-003 — Deterministic Engine
**Source:** RFC §4.5.2  
**Description:** Given the same sequence of operations, the engine must always produce the same state. There must be no non-deterministic behavior (no random IDs generated inside apply, no timestamps used as keys).  
**Priority:** Critical

---

### REQ-004 — Operation Idempotence
**Source:** RFC §4.4.4, §4.5.2  
**Description:** Applying the same operation twice (same `op.id`) must not duplicate effects. The engine tracks applied operation IDs.  
**Priority:** Critical

---

### REQ-005 — Dynamic Schema
**Source:** RFC §3 Principles, §4.4.2 (SchemaOperation)  
**Description:** The table schema (columns) can change at runtime via `add_column`, `remove_column`, `reorder_columns` operations. Schema changes must increment `schema.version`.  
**Priority:** High

---

### REQ-006 — Mandatory Virtualization
**Source:** RFC §3 Principles, Annex A §2  
**Description:** The UI layer must virtualize row and column rendering. Virtualization is a hard architectural constraint, not an optimization. Required for datasets > 1000 rows. Target: 50k rows in MVP, 100k–500k in target.  
**Priority:** Critical

---

### REQ-007 — Derived Loading State
**Source:** RFC §3 Principles, §4.4.3, §4.4.6  
**Description:** Loading/pending state must NOT be stored in row or cell data. It must be derived from the set of pending operations indexed by `TargetDescriptor`. The store/manager must expose `getPendingByTarget(target)`.  
**Priority:** High

---

### REQ-008 — Optimistic Updates
**Source:** RFC §4.4.5, §4.4.7  
**Description:** When a user action creates an operation, the engine applies it immediately (optimistic). The UI reflects the new value while the operation is in `pending`/`applying` state. On backend confirmation, the loading indicator is removed.  
**Priority:** High

---

### REQ-009 — Batching and Debounce
**Source:** RFC §4.4.6, §4.4.9  
**Description:** The OperationManager must batch operations before sending to backend. Recommended debounce window: 50–100ms. Recommended batch size: 50–500 ops.  
**Priority:** High

---

### REQ-010 — Backend Reconciliation
**Source:** RFC §4.4.7, §4.6  
**Description:** The backend is the source of truth. It may accept (ack), reject, or transform operations. The client must reconcile its state with server responses. Three strategies are supported: Replace, Transform, Reject.  
**Priority:** High

---

### REQ-011 — Operation State Machine
**Source:** RFC Annex A §1  
**Description:** Each operation has a lifecycle: `pending` → `applying` → (`confirmed` | `error`). The OperationManager maintains this state per operation.  
**Priority:** High

---

### REQ-012 — REST Transport (MVP)
**Source:** RFC §4.4.8, §4.6.1  
**Description:** The MVP transport is `POST /operations` accepting `Operation[]` and returning `{ ack, errors?, serverOps? }`. Initial state is loaded via `GET /table`.  
**Priority:** High (MVP)

---

### REQ-013 — WebSocket Transport (Future)
**Source:** RFC §4.4.8, §4.6.5  
**Description:** Future phases support WebSocket for streaming server-driven operations. The `OperationTransport` interface must be designed to support both HTTP and WebSocket adapters interchangeably.  
**Priority:** Low (post-MVP)

---

### REQ-014 — Transport Abstraction
**Source:** RFC Annex C §4.0  
**Description:** The `OperationManager` must depend on an `OperationTransport` interface, not on a concrete HTTP/WebSocket implementation. Adapters are provided as separate packages.  
**Priority:** High

---

### REQ-015 — Stable Row and Column IDs
**Source:** RFC §9 Open Questions  
**Description:** Rows and columns must be identified by stable string IDs (e.g., backend-assigned UUIDs). Positional indexes must not be used as identifiers. IDs are required for scroll anchoring and delta reconciliation.  
**Priority:** Critical

---

### REQ-016 — Monorepo Structure
**Source:** RFC Annex C §2  
**Description:** The codebase must be organized as a monorepo with the following top-level packages: `core`, `operations`, `store`, `react`, `ui` (optional), `api-client` (optional). A `playground` app in `/apps` is included for development and demos.  
**Priority:** High

---

### REQ-017 — Schema Version Tracking
**Source:** RFC §4.4.4, §4.5.4  
**Description:** Every `SchemaOperation` must increment `schema.version`. Clients must detect schema version mismatches with the backend and trigger a full refresh.  
**Priority:** High

---

### REQ-018 — Lazy Cell Resolution
**Source:** RFC §4.5.5  
**Description:** Cells not yet populated return a default value (null or column default) without requiring backfill. New columns do not require immediate backfill of existing rows.  
**Priority:** Medium

---

### REQ-019 — Bulk and Range Operations
**Source:** RFC §4.4.2, §6 Features  
**Description:** The system must support `bulk_update` (multiple cells in arbitrary positions), `update_column` (one value per row for a given column), and range-based target descriptors for clipboard paste scenarios.  
**Priority:** High

---

### REQ-020 — Scroll Stability on Row Insertion
**Source:** RFC §9.1 Case 6  
**Description:** When rows are inserted while the user is scrolled, the UI must capture a scroll anchor before applying, recompute layout, and restore the anchor to prevent viewport jumping.  
**Priority:** Medium

---

### REQ-021 — Error Handling per Operation
**Source:** RFC §4.6.1, §9.1 Case 5  
**Description:** When the backend rejects an operation, the OperationManager calls `fail(opId)`. The engine may optionally revert the optimistic change. The UI must show an error state for the affected target.  
**Priority:** High

---

### REQ-022 — Performance Targets
**Source:** RFC Annex A §2  
**Description:** The system must maintain 60fps rendering with individual re-renders < 16ms. Operations must be indexed by target for O(1) pending-state lookups (no O(n) scans per render).  
**Priority:** High

---

### REQ-023 — Out-of-Order Operation Buffering
**Source:** RFC §4.4.10  
**Description:** Operations that reference columns or rows not yet present in the schema must be buffered until the target entity is registered. Ghost entities must not crash the engine.  
**Priority:** Medium

---

### REQ-024 — Security — Server-Side Validation
**Source:** RFC §4.6.7, Annex B §7  
**Description:** The backend must never trust client-computed values. All operations must be revalidated server-side for type correctness, constraint satisfaction, and sanitization.  
**Priority:** Critical

---

### REQ-025 — Observability
**Source:** RFC Annex A §5  
**Description:** The system should expose hooks for operation logging, pending count metrics, latency tracking, and desync detection. This is recommended but not blocking for MVP.  
**Priority:** Low

## 4. Open Questions

| ID | Question | Source | Impact |

## 5. Inferred Assumptions (for unblocking generation)

| ID | Assumption | Rationale |
|----|-----------|-----------|
| AS-01 | MVP excludes formulas and real-time collaboration | RFC §9 explicitly lists these as optional / post-MVP |
| AS-02 | Client generates operation IDs (e.g., nanoid/UUID) | Standard optimistic pattern; backend revalidates |
| AS-03 | Schema dynamism level is "Intermediate" for MVP: add/remove columns supported, type changes deferred | Most common use case; keeps complexity manageable |
| AS-04 | Operation tracking and derived loading state is in scope for MVP | Required for correct UX in optimistic update pattern |
| AS-05 | REST transport (HTTP) is the MVP; WebSocket is future | Explicitly stated in RFC |
| AS-06 | Row count worst-case for MVP is 50k rows | RFC Annex A §2 explicitly sets this |
| AS-07 | Auto-revert on operation error is the default policy | Keeps state consistent without complex manual resolution UI |
