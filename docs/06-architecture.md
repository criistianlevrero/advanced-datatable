## 1. Architectural Pattern

The system uses a **layered + pipeline architecture** with a **unidirectional data flow** pattern:

- Mutations only flow downward through the pipeline: UI вҶ’ Dispatch вҶ’ OperationManager вҶ’ Engine
- State reads flow upward: Engine вҶ’ Store вҶ’ React Hooks вҶ’ UI
- The backend is an external authority; its responses flow back through the OperationManager
- No direct coupling between non-adjacent layers

This is complemented by **Domain-Driven Design** principles for intra-package organization (schema, data, engine as separate bounded contexts), and **Inversion of Control** for the transport layer.

```

        в”Ӯ
```
```
        в–ј

        в”Ӯ
```

## 7. Key Architectural Decisions

### AD-01: Engine has zero React dependencies
**Decision:** `packages/core` and `packages/operations` have zero React/Zustand dependencies.  
**Rationale:** Enables testing in Node.js, reuse in non-React environments, and prevents UI framework
updates from breaking core business logic.  
**Constraint:** All reactivity plumbing lives in `packages/store` (Zustand).

---

### AD-02: Operations are immutable values
**Decision:** Operations are plain objects. Once created, they are not mutated.  
**Rationale:** Enables safe serialization, logging, and replay. Avoids subtle bugs from
shared mutable references.

---

### AD-03: Derived loading state (not data flags)
**Decision:** Loading/pending state is derived from the `TargetIndex` in `OperationManager`, not stored in Row or Cell.  
**Rationale:** Keeps data model clean. Avoids stale state after background ops resolve. Enables O(1) lookups.

---

### AD-04: OperationTransport is injected (not imported)
**Decision:** `OperationManagerImpl` accepts `IOperationTransport` via constructor.  
**Rationale:** Enables swapping HTTP for WebSocket without modifying core logic. Supports mocking in tests.

---

### AD-05: Store is instantiated per table (factory)
**Decision:** `createTableStore()` returns a new store per call. No global singleton.  
**Rationale:** Multiple DataTable instances must coexist on the same page without shared state.

---

### AD-06: Virtualization is mandatory from the start
**Decision:** The grid always uses TanStack Virtual for row (and optionally column) rendering.  
**Rationale:** 50k row MVP target makes non-virtualized rendering unacceptable. Retrofitting virtualization
into an existing non-virtualized renderer is very expensive.

---

### AD-07: Schema version is the desync detector
**Decision:** Every`SchemaOperation` increments `schema.version`. Client checks version against backend on load.  
**Rationale:** Simple, deterministic, low-overhead way to detect schema desync without polling entire state.
