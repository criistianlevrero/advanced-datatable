## 2. Domain Descriptions

### 2.1 Domain: SCHEMA

**Bounded Context:** Represents the structural definition of the table.

**Responsibilities:**
- Define what columns exist, their types, display names, widths, and metadata
- Maintain column ordering (`columnOrder: string[]`)
- Track schema version (incremented on every structural change)
- Validate that a column exists before an operation references it

**Core Entities:**
- `ColumnSchema`
- `TableSchema`

**Business Rules:**
- Column IDs must be unique within a schema
- `schema.version` must increment on every `SchemaOperation`
- Column type cannot be changed without an explicit operation (future: `update_column_type`)
- Column order is always represented as a stable ordered array of IDs

**Package:** `packages/core/src/models/`

---

### 2.2 Domain: DATA

**Bounded Context:** Represents the actual row/cell data stored in the table.

**Responsibilities:**
- Store rows as a Map keyed by stable row ID
- Store cells as a record keyed by column ID within each row
- Maintain row ordering (`rowOrder: string[]`)
- Resolve cell values with lazy defaults (null or column-defined default)

**Core Entities:**
- `Row`
- `Cell`
- `TableState`

**Business Rules:**
- Row IDs must be stable (UUID, backend-assigned)
- Cell value is `any` at transport level; semantic validation is backend responsibility
- Cells that don't exist resolve to `{ value: null }` or column default
- Row order is always an explicit ordered array of IDs (not positional)

**Package:** `packages/core/src/models/`

---

### 2.3 Domain: ENGINE

**Bounded Context:** The deterministic state machine that applies operations to produce a new table state.

**Responsibilities:**
- Apply typed operations to mutate Schema and Data domains
- Guarantee determinism: same ops sequence → same state
- Guarantee idempotence: applying the same `op.id` twice has no extra effect
- Track which operation IDs have been applied (dedup set)
- Handle lazy cell creation (upsert on `set_cell`)
- Buffer operations that reference non-existent entities (policy-driven)

**Core Entities:**
- `TableEngine` (interface)
- `TableEngineImpl` (class)

**Business Rules:**
- No IO, no side effects inside `apply()`
- No timestamps or random values generated inside engine
- `SchemaOperation` always increments `schema.version`
- `applyBatch(ops)` = sequential `apply()` per op; same guarantees

**Package:** `packages/core/src/engine/`

---

### 2.4 Domain: OPERATION

**Bounded Context:** The lifecycle management and orchestration of operations.

**Responsibilities:**
- Create and register operations with `pending` status
- Maintain a set of in-flight operations indexed by `op.id`
- Index operations by `TargetDescriptor` for O(1) pending-state lookups
- Execute the pipeline: optimistic apply → batch → send → receive ack/error → reconcile
- Batch and debounce outgoing operations (50–100ms window)
- Retry logic (policy-configurable)
- Transition operation lifecycle states: pending → applying → confirmed | error

**Core Entities:**
- `OperationManager` (interface)
- `OperationManagerImpl` (class)
- `TargetIndex` (internal index structure)

**Business Rules:**
- A `pending` op must always be applied optimistically before sending
- An op in `confirmed` state cannot transition back
- An op in `error` state triggers optional revert in the engine
- Batching window must not exceed 100ms for acceptable UX
- Duplicate `op.id` submissions to the manager are no-ops

**Package:** `packages/operations/src/`

---

### 2.5 Domain: TRANSPORT

**Bounded Context:** The abstraction layer for network communication between client and backend.

**Responsibilities:**
- Abstract HTTP (REST) and WebSocket implementations behind a common interface
- Send batches of operations to the backend
- Receive acknowledgements, errors, and server-initiated operations
- Provide a `subscribe` mechanism for server push (WebSocket)

**Core Entities:**
- `OperationTransport` (interface)
- `HttpTransport` (adapter, in `api-client` package)
- `WebSocketTransport` (adapter, future)

**Business Rules:**
- The `OperationManager` must only depend on `OperationTransport`, not on concrete adapters
- The transport is responsible for serialization/deserialization of operation payloads
- REST MVP: `POST /operations` + `GET /table`
- The transport must never modify operation payloads

**Package:** `packages/api-client/src/` (optional/adapter)

---

### 2.6 Domain: STORE

**Bounded Context:** The reactive state bridge between the Engine/OperationManager and the UI.

**Responsibilities:**
- Wire `TableEngine` and `OperationManager` into a Zustand store
- Expose reactive getters for schema, rows, rowOrder, and pending targets
- Trigger re-renders on relevant state changes
- Avoid exposing raw mutable engine state directly to UI

**Core Entities:**
- `TableStore` (type)
- `createTableStore()` (factory)

**Business Rules:**
- All mutations must go through `OperationManager.apply()`; no direct engine calls from UI
- Derived state (e.g., loading per cell) must be computed from pending ops, not stored
- Store must be instantiable per table instance (no global singleton)

**Package:** `packages/store/src/`

---

### 2.7 Domain: UI / REACT

**Bounded Context:** Presentation layer: hooks, components, and virtualization.

**Responsibilities:**
- Provide React hooks (`useDataTable`, `useCell`, `useSelection`) as the public API for React consumers
- Render the virtualized grid using TanStack Table + TanStack Virtual
- Handle user interactions (cell edit, selection, clipboard paste) and convert them into operations
- Display loading and error states derived from the Store

**Core Entities:**
- `useDataTable` hook
- `useCell` hook
- `useSelection` hook
- `DataTable` component (optional UI package)
- `Cell` renderer (optional UI package)
- `Grid` layout (optional UI package)

**Business Rules:**
- Hooks must not call engine directly — they dispatch through the store
- Components must be memoized to prevent unnecessary re-renders
- Virtualizer window is the source of truth for which rows/columns to render

**Packages:** `packages/react/src/`, `packages/ui/src/` (optional)

## 3. Inter-Domain Interfaces

| From | To | Interface / Event | Description |
|------|----|-------------------|-------------|
| UI | STORE | `store.dispatch(op)` | User action converted to operation |
| STORE | OPERATION | `manager.apply(op)` | Store delegates operation execution |
| OPERATION | ENGINE | `engine.apply(op)` | Optimistic application |
| OPERATION | TRANSPORT | `transport.send(ops[])` | Batch transmission |
| TRANSPORT | OPERATION | `{ ack, errors, serverOps }` | Backend response |
| OPERATION | ENGINE | `engine.apply(serverOp)` | Server corrections applied |
| ENGINE | STORE | State change notification | Zustand subscription triggers re-render |
| STORE | UI | Derived state (cells, schema, pending) | React hooks read from store |
| SCHEMA | ENGINE | Structural read | Engine reads/writes schema during apply |
| DATA | ENGINE | Row/Cell read-write | Engine reads/writes rows during apply |

## 4. Domain Dependencies (Direction)

```
UI/React  →  Store  →  Operation  →  Engine  →  Schema
                    →                         →  Data
                    →  Transport  (→ Backend)
```

- **UI** depends on Store (hooks)
- **Store** depends on Operation + Engine (wires them)
- **Operation** depends on Engine (apply) + Transport (send)
- **Engine** depends on Schema + Data (in-memory)
- **Transport** has no upstream dependencies (it is a leaf I/O adapter)

**Inversion of control:**  
`Operation` does not import concrete Transport — it receives an `OperationTransport` via constructor injection.
